# Plan: Frontend Dashboard + Chat for Date Ideas

> Generated by /plan on 2026-04-09 | Approach: Pragmatic Next.js Migration

## Feature Summary

Migrate the existing project (Telegram bot + Google Sheets + Gemini AI) to a Next.js App Router app that coexists with the Telegram webhook. The frontend includes two main features:

1. **Dashboard** — Filterable and searchable list view of all date ideas, with full CRUD operations (add, edit, delete).
2. **Chat** — Single-turn chat interface where the user talks to the Gemini agent to add and query date ideas, replicating the Telegram bot experience in the browser.

Authentication is a simple shared password (couple's PIN) protected via Next.js middleware with an httpOnly cookie. The backend remains Google Sheets (with a Postgres migration planned as a future issue).

## Functional Requirements

| Requirement | Trigger | Expected Behavior | Timing | Reversible? | Accessibility |
|-------------|---------|-------------------|--------|-------------|---------------|
| Auth gate | Request to `/dashboard` or `/chat` | Redirect to `/login` if no valid cookie | Immediate (middleware) | Yes — login restores access | Form with associated labels, autofocus |
| Login | Submit form with password | POST `/api/auth/login`; set httpOnly cookie; redirect to `/dashboard` | < 100ms | Yes — logout clears cookie | Error message in Spanish on failure |
| List ideas | Page load `/dashboard` | Server Component fetch + client-side filterable list | Server render + instant filters | n/a | Items as `<article>` elements with headings |
| Filter/search | Dropdown/input change | Client-side filter of in-memory array (max 100 rows) | Instant | Yes — clear filter | Controls with associated `<label>` |
| Add idea | Submit form `/dashboard/nueva` | POST `/api/ideas` → `parseAddMessage()` + `appendIdea()` | 1–5s (Gemini) | No (but delete exists) | Loading spinner + submit disabled |
| Edit idea | Submit form `/dashboard/[id]` | PATCH `/api/ideas/[id]` → `updateIdea()` | < 1s (Sheets write) | Yes — can re-edit | Form pre-filled from server props |
| Delete idea | Delete button on IdeaCard | Confirm dialog → DELETE `/api/ideas/[id]` → `router.refresh()` | < 1s | No | Dialog with trapped focus |
| Chat query | Submit message on `/chat` | POST `/api/chat` → `answerQuery()` → Spanish response | 2–8s (Gemini) | n/a | `aria-live="polite"` for response |

## Edge Cases

### Auth
- [ ] `FRONTEND_PASSWORD` not set → middleware catch → redirect to `/login` with `?error=config` param
- [ ] Tampered cookie value → middleware rejects, redirect to `/login`
- [ ] Multiple tabs → share cookie; logout in one leaves others stale until navigation
- [ ] Empty password in form → client-side validation, don't send request

### Data & Content
- [ ] `deleteIdea()` race condition (2 simultaneous deletes) → acceptable for 2 users; document with code comment
- [ ] `getAllIdeas()` 100-row cap → show visible counter "Showing X ideas" in UI
- [ ] ID not found on update/delete → API returns 404 with Spanish message
- [ ] Gemini returns invalid JSON on add → existing fallback in `parseAddMessage()` uses raw input as title
- [ ] Gemini timeout → `maxDuration = 55` in route files; UI shows error after timeout

### Responsive
- [ ] Mobile (< 640px) — cards stack vertically, filters collapsed, nav as simple header
- [ ] Tablet (640-1024px) — 2-column card grid
- [ ] Desktop (> 1024px) — 3-column grid, filters visible in sidebar or inline

### Performance
- [ ] `getAllIdeas()` on every page load → one Sheets API round-trip per load; acceptable for personal app
- [ ] Client-side filtering of 100 rows → trivially fast, no concern
- [ ] Gemini calls are slow (1-8s) → clear loading states in UI

### tsconfig Migration
- [ ] `api/webhook.ts` with Node.js types → verify with `bun run build` after migration
- [ ] `googleapis` Node.js-only → `serverExternalPackages` in `next.config.ts`; never import in Client Components
- [ ] `src/config.ts` imported at module level → safe for Server Components and route handlers; dangerous in Client Components

## Constraints

- Google Sheets as backend: no index by ID, update/delete require full read
- All agent responses are in Spanish (system prompts)
- `googleapis` incompatible with Edge Runtime — all functions must run on Node.js runtime
- Telegram webhook (`api/webhook.ts`) must continue working unchanged
- Do not import anything from `src/bot/` in web code (eagerly loads `TELEGRAM_BOT_API_KEY`)
- `maxDuration = 55` on routes that call Gemini
- No test runner — verification is manual
- The `date-manager-app-5902d1974c12.json` file (service account) must not be committed

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| tsconfig migration breaks webhook | High — Telegram bot stops working | Verify with `bun run build` + preview deploy before production; webhook doesn't use ESM imports |
| `googleapis` imported in Client Component | High — runtime crash | Keep imports only in `src/services/sheets.ts`; never import sheets.ts from `"use client"` components |
| Race condition on simultaneous delete | Low — only 2 users | Document; acceptable for personal scale |
| Gemini timeout on web requests | Medium — bad UX | `maxDuration = 55` + loading states + error handling in UI |
| Cookie session = plaintext password | Low — personal couple's app | httpOnly + secure flags; document limitation |
| Google Sheets rate limiting | Low — few requests | Acceptable; future: migrate to Postgres (#2) |
| Next.js build ignores `api/webhook.ts` | Medium — might not deploy | Vercel treats root `api/` as legacy standalone functions separate from Next.js; verify in preview |
