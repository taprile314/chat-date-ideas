# Plan: Vercel Durable Workflows + DurableAgent Chatbot

> Generated by /plan on 2026-04-12 | Approach: Original DurableAgent (WDK beta)

## Feature Summary

Migrate both the web chatbot and Telegram bot to use Vercel Durable Workflows with `DurableAgent` from `@workflow/ai/agent`. The agent has 4 tools ‚Äî `searchIdeas`, `addIdea`, `listAllIdeas`, `getRandomIdea` ‚Äî backed by the existing Google Sheets data store. The web path streams via `WorkflowChatTransport`; the Telegram path fires the agent asynchronously after immediately acking the webhook with HTTP 200. This approach was chosen despite the WDK packages being in beta because the user explicitly wants the full durable workflow pattern as demonstrated in the `vercel/workflow-examples/rag-agent` template.

**Why DurableAgent over stable ToolLoopAgent:** The user chose this approach for durable workflow semantics (`'use workflow'`, `'use step'`), which provide automatic retry, state persistence across step boundaries, and the architectural pattern from the official Vercel template. The trade-off is beta package risk, which is mitigated by version pinning and a documented rollback path.

## Edge Cases

### Data & Content
- [ ] Empty ideas list ‚Äî each tool returns "No hay ideas guardadas" string; agent relays without hallucinating
- [ ] `parseAddMessage` fallback ‚Äî existing try/catch returns raw input as title; addIdea step must not re-throw
- [ ] Zero matching search results ‚Äî tool returns descriptive message, agent suggests broadening criteria
- [ ] Very long ideas list ‚Äî at ~100 ideas, full list fits in tool response; no pagination needed

### Telegram-Specific
- [ ] 5-second ack window ‚Äî webhook returns 200 immediately, agent runs asynchronously (fire-and-forget)
- [ ] Duplicate webhook retries ‚Äî once 200 is sent, Telegram won't retry; pattern prevents duplicates
- [ ] 4096-char message limit ‚Äî `sendReply` truncation preserved from current code

### Vercel Workflows / WDK
- [ ] Beta package stability ‚Äî pin exact versions (`workflow@4.2.0-beta.67`, `@workflow/ai@4.0.1-beta.54`)
- [ ] WDK compiler plugin + Turbopack ‚Äî test build in Phase 0; fallback: disable Turbopack
- [ ] `googleapis` in `serverExternalPackages` ‚Äî verify externalization works in workflow step bundles
- [ ] Module-level Sheets auth client across step boundaries ‚Äî verify auth client survives step re-hydration; fallback: lazy-init per step
- [ ] `@ai-sdk/google` model with DurableAgent ‚Äî verify Gemini model works; test first tool call before full wiring

### Performance
- [ ] Cold start amplification ‚Äî workflow runtime adds second cold start path; acceptable for couple's app
- [ ] `getAllIdeas()` per tool invocation ‚Äî acceptable at ~100 ideas scale
- [ ] Cleanup ‚Äî no persistent connections; JWT auth client is stateless per request

### Accessibility
- [ ] `aria-live="polite"` on message container preserved
- [ ] Stop button during streaming preserved
- [ ] All UI text remains in Spanish

## Constraints

- Package manager and runtime: bun
- Model: Gemini 2.5 Flash via `@ai-sdk/google` with `GEMINI_API_KEY`
- All AI responses in Spanish ‚Äî system prompts enforce this
- `src/services/sheets.ts` server-only (`googleapis` in `serverExternalPackages`)
- Do not import from `src/bot/` in web/Next.js code
- `maxDuration = 55` on all routes calling Gemini or running workflows
- `workflow` and `@workflow/ai` pinned to exact beta versions
- Telegram webhook must respond 200 within 5 seconds ‚Äî agent runs fire-and-forget

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| `workflow`/`@workflow/ai` beta breaking change | Agent stops working, 9+ file rollback | Pin exact versions; document rollback to `streamQueryResponse` (1-file revert in `ai.ts`) |
| WDK compiler plugin + Turbopack incompatible | Build fails, can't deploy | Test in Phase 0 before any logic changes; fallback: remove `--turbopack` from dev script |
| `DurableAgent` incompatible with `@ai-sdk/google` | Model calls fail at runtime | Test first tool call in Phase 2 before wiring routes; fallback: use OpenAI-compatible model |
| `WorkflowChatTransport` different message protocol than `DefaultChatTransport` | Frontend silently breaks | Test `message.parts` rendering immediately in Phase 4; fallback: keep `DefaultChatTransport` with modified route |
| Telegram 5-second ack timeout breach | Duplicate messages, duplicate idea additions | Fire-and-forget pattern: respond 200 immediately, process agent asynchronously |
| `googleapis` not externalized in workflow step bundles | Runtime import error for Sheets | Verify `serverExternalPackages` applies to workflow bundles; fallback: add `googleapis` to WDK external config |
| Module-level Sheets auth client fails across step re-hydration | Auth errors in tool steps | Verify with first Sheets tool call; fallback: lazy-init auth client inside each step function |
| Cold start amplification (workflow runtime + serverless function) | Slow first response on inactive app | Acceptable for couple's app usage pattern; no mitigation needed |

## Rollback Path

If the WDK approach fails at any phase, the rollback is:

1. Remove `workflow` and `@workflow/ai` from `package.json`
2. Revert `next.config.ts` (remove WDK plugin)
3. Revert `app/api/chat/route.ts` to `streamQueryResponse` call
4. Revert `app/_components/ChatBox.tsx` to `DefaultChatTransport`
5. Delete `workflows/` directory, `app/api/agent/route.ts`, `src/bot/agent-client.ts`
6. Revert `api/webhook.ts` and `src/bot/router.ts`
7. Run `bun install && bun run build`

The current working architecture on `feat/ai-sdk-migration` branch serves as the known-good baseline.
